**Researched:** 2026-02-27

**Confidence:** HIGH

## Summary

Convex officially supports self-hosting via Docker Compose since February 2025, using the same codebase as their cloud-hosted service. The official stack consists of three services: the Convex backend (Rust + TypeScript), the Convex dashboard (Next.js), and a persistent data layer (SQLite by default, with Postgres/MySQL support).

Coolify has official documentation for deploying Convex using its Docker Compose build pack. The deployment strategy involves:
1. Creating a Docker Compose file with Convex backend and dashboard services
2. Deploying via Coolify's Docker Compose build pack
3. Generating admin keys via Coolify's terminal
4. Configuring frontend environment variables to point to the self-hosted backend

**Primary recommendation:** Use Coolify's Docker Compose build pack with Convex's official docker-compose.yml, pointing the frontend to the self-hosted deployment via `CONVEX_SELF_HOSTED_URL` and `CONVEX_SELF_HOSTED_ADMIN_KEY` environment variables.

## Architecture Patterns

### Recommended Deployment Structure

```
coolify-convex-deployment/
├── docker-compose.yml    # Convex services (backend + dashboard)
├── .env                  # Admin key (NOT in version control)
└── (frontend app - separate Coolify resource)
```

### Pattern 1: Coolify Docker Compose Deployment

**What:** Deploy Convex backend and dashboard as a service stack in Coolify using the official docker-compose.yml.

**When to use:** Production deployments on existing Coolify infrastructure.

**Example:**
```yaml
# docker-compose.yml (from Convex official repo)
services:
  backend:
    image: ghcr.io/get-convex/convex-backend:latest
    ports:
      - "3210:3210"  # Convex backend API
      - "3211:3211"  # HTTP actions
    volumes:
      - data:/convex/data
    environment:
      - CONVEX_CLOUD_ORIGIN=${CONVEX_CLOUD_ORIGIN}
      - POSTGRES_URL=${POSTGRES_URL}  # Optional: for Postgres
    healthcheck:
      test: curl -f http://localhost:3210/version
      interval: 5s
      start_period: 10s

  dashboard:
    image: ghcr.io/get-convex/convex-dashboard:latest
    ports:
      - "6791:6791"
    environment:
      - NEXT_PUBLIC_DEPLOYMENT_URL=${CONVEX_CLOUD_ORIGIN}
    depends_on:
      backend:
        condition: service_healthy

volumes:
  data:
```

**Source:** https://github.com/get-convex/convex-backend/blob/main/self-hosted/docker/docker-compose.yml

### Pattern 2: Frontend Environment Configuration

**What:** Configure the Vite frontend to connect to the self-hosted Convex backend.

**When to use:** All deployments using self-hosted Convex.

**Example:**
```bash
# .env.production (for Vite build)
VITE_CONVEX_URL=https://convex.your-domain.com
CONVEX_SELF_HOSTED_URL=https://convex.your-domain.com
CONVEX_SELF_HOSTED_ADMIN_KEY=<generated_admin_key>
```

**Source:** https://docs.convex.dev/client/react/deployment-urls

### Pattern 3: Admin Key Generation

**What:** Generate admin keys via Coolify terminal for dashboard access and CLI deployment.

**When to use:** Initial setup and after any credential rotation.

**Example:**
```bash
# Via Coolify terminal (per official Coolify docs)
docker compose exec backend ./generate_admin_key.sh
# Output: convex-self-hosted|01xxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx
```

**Source:** https://coolify.io/docs/services/convex

### Anti-Patterns to Avoid

- **Hardcoding admin keys in docker-compose.yml:** Always use environment variables via Coolify's env management
- **Using 127.0.0.1 for CONVEX_CLOUD_ORIGIN:** Use actual host IP or domain for external access
- **Skipping health checks:** Backend healthcheck prevents dashboard startup race conditions
- **Not configuring persistent volumes:** Data loss on container restart without volume mounts

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Convex deployment orchestration | Custom Docker setup | Coolify Docker Compose build pack | Official integration, handles SSL, health checks, persistent storage |
| Admin key generation | Custom key generation script | `./generate_admin_key.sh` in backend container | Official method, cryptographically secure |
| Database persistence | Custom backup scripts | Coolify persistent volumes or Postgres service | Built-in volume management, automatic backups |

**Key insight:** Convex self-hosting is intentionally designed as a Docker Compose deployment. Hand-rolling infrastructure around it introduces unnecessary complexity and maintenance burden.

## Common Pitfalls

### Pitfall 1: Wrong CONVEX_CLOUD_ORIGIN Format
**What goes wrong:** Frontend cannot connect to backend, connection refused errors.

**Why it happens:** Using `127.0.0.1` or `localhost` for remote deployments instead of actual host IP/domain.

**How to avoid:** Always use the actual hostname or IP address for `CONVEX_CLOUD_ORIGIN` in production. For Coolify, use the assigned domain.

**Warning signs:** Connection refused errors in browser console, frontend shows "not connected" status.

### Pitfall 2: Missing Admin Key After Deployment
**What goes wrong:** Cannot access dashboard or deploy Convex functions.

**Why it happens:** Admin key generation step is manual via terminal, not automated.

**How to avoid:** Document admin key generation immediately after deployment in secure secrets manager. Run `./generate_admin_key.sh` via Coolify terminal.

**Warning signs:** Dashboard authentication fails, `npx convex deploy` prompts for URL/key.

### Pitfall 3: Volume Not Persisted
**What goes wrong:** All Convex data lost on container restart or redeployment.

**Why it happens:** Forgetting to configure persistent storage in Coolify for the `data` volume.

**How to avoid:** Always configure Coolify persistent storage for the `data` volume defined in docker-compose.yml.

**Warning signs:** Data disappears after Coolify service restart.

### Pitfall 4: Postgres Connection String Format
**What goes wrong:** Convex fails to connect to Postgres database.

**Why it happens:** Convex requires specific Postgres URL format without database name or extra params.

**How to avoid:** Strip connection string to `postgres://user:pass@host` format (no database name, no `?sslmode=require`).

**Warning signs:** Backend logs show database connection errors.

```typescript
// Source: https://docs.convex.dev/client/react/deployment-urls
import { ConvexProvider, ConvexReactClient } from "convex/react";

const convexUrl = import.meta.env.VITE_CONVEX_URL;
const client = new ConvexReactClient(convexUrl, {
  skipConvexDeploymentUrlCheck: true, // Required for self-hosted with custom domain

  return (
    <ConvexProvider client={client}>
      {/* Your app */}
    </ConvexProvider>
  );
}
```

## Open Questions

1. **Coolify Domain Configuration**
   - What we know: Coolify can assign domains to Docker Compose services
   - What's unclear: Exact domain format for CONVEX_CLOUD_ORIGIN in Coolify environment
   - Recommendation: Test with assigned Coolify domain first, use custom domain if needed

2. **Postgres vs SQLite for Initial Deployment**
   - What we know: SQLite is default, Postgres supported via POSTGRES_URL
   - What's unclear: Whether Nexus production workload requires Postgres immediately
   - Recommendation: Start with SQLite, migrate to Postgres if scale demands it

3. **Multi-environment Strategy**
   - What we know: Coolify supports multiple environments (development, production)
   - What's unclear: Whether to run separate Convex deployments per environment
   - Recommendation: Single Convex deployment for MVP, separate staging if team grows
