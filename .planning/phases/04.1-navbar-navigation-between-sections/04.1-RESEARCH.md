## Summary

Phase 4.1 replaces the current flat `AppRoutes` + inline `SyncComponents` header with a proper layout shell that renders a persistent responsive navbar. The existing routing infrastructure uses React Router v7 (library mode, not framework mode) with `<Routes>` and `<Route>` components. The correct pattern is to introduce a `<AppLayout>` component that uses `<Outlet />` for child routes, or to keep the current `<Routes>` structure and wrap it in a layout div — the former (Outlet-based nested routing) is the idiomatic React Router v7 approach and is recommended.

The current codebase has three hard-coded section routes (`/admin/builder`, `/worker/forms`, `/reviewer/dashboard`) plus role-based redirect logic in `ProtectedRoute`. The nav shell needs to read the user's role from the existing `useRole()` hook (already in `AuthContext`) to determine which nav items to show. User identity (name + avatar) must come from Clerk's `useUser()` hook (`user.firstName`, `user.lastName`, `user.imageUrl`) — this is a new hook call; it is NOT currently in `AuthContext`.

The Tailwind + shadcn/ui stack is already configured. The existing shadcn `cn()` utility, `lucide-react` icons, and the color tokens (`bg-background`, `border`, `muted`, `primary`, etc.) are all available. No new packages are required for this phase.

**Primary recommendation:** Introduce a single `AppLayout` component that renders the responsive nav shell (sidebar on desktop, bottom tabs + top bar on mobile) around an `<Outlet />`. Refactor `AppRoutes` to use nested routes with `AppLayout` as the parent. Role-based nav items are derived from `useRole()`. User identity from `useUser()` (Clerk). No new dependencies.

### Pattern 1: Outlet-Based Layout Shell

**What:** A parent route component that renders persistent chrome (sidebar/bottom tabs) and an `<Outlet />` for child content. React Router renders the matched child route into the Outlet.

**When to use:** Any time a set of routes shares a persistent layout (nav, header, footer). This is the idiomatic React Router v7 library-mode pattern.

**Example:**
```typescript
// Source: Context7 /remix-run/react-router — Outlet Component
import { Outlet, useLocation, Link } from "react-router";

function AppLayout() {
  return (
    <div className="flex min-h-screen">
      {/* Desktop sidebar - hidden on mobile */}
      <aside className="hidden md:flex flex-col w-64 border-r bg-background">
        <nav>{/* nav items */}</nav>
        <div className="mt-auto">{/* user identity */}</div>
      </aside>

      {/* Main content */}
      <div className="flex-1 flex flex-col">
        {/* Mobile top bar - hidden on desktop */}
        <header className="md:hidden border-b bg-background px-4 py-3 flex items-center justify-between">
          {/* page title + user avatar */}
        </header>

        {/* Desktop top header - hidden on mobile */}
        <header className="hidden md:flex border-b bg-background px-4 py-3 items-center justify-between">
          <h1 className="text-xl font-bold">Nexus QC Forms</h1>
          {/* SyncIndicator */}
        </header>

        <main className="flex-1 overflow-auto pb-16 md:pb-0">
          <Outlet />
        </main>

        {/* Mobile bottom tab bar - hidden on desktop */}
        <nav className="md:hidden fixed bottom-0 left-0 right-0 border-t bg-background">
          {/* bottom tabs */}
        </nav>
      </div>
    </div>
  );
}
```

### Pattern 2: Refactored AppRoutes with Nested Routes

**What:** Wrap authenticated routes inside a parent `<Route element={<AppLayout />}>` so the layout renders once and child routes swap into `<Outlet />`.

**When to use:** When the layout applies to all protected routes (which it does here).

**Example:**
```typescript
// Refactored src/routes/index.tsx
import { Routes, Route, Navigate } from "react-router";
import AppLayout from "../components/layout/AppLayout";

export default function AppRoutes() {
  return (
    <ProtectedRoute>
      <Routes>
        <Route path="/sign-in" element={<SignInPage />} />
        <Route element={<AppLayout />}>
          <Route path="/" element={<Navigate to="/admin/builder" replace />} />
          <Route path="/admin/builder" element={<AdminRoute><AdminBuilder /></AdminRoute>} />
          <Route path="/worker/forms" element={<WorkerRoute><FormFillingPage /></WorkerRoute>} />
          <Route path="/reviewer/dashboard" element={<ReviewerRoute><ReviewerDashboard /></ReviewerRoute>} />
          <Route path="*" element={<Navigate to="/admin/builder" replace />} />
        </Route>
      </Routes>
    </ProtectedRoute>
  );
}
```

### Pattern 3: Role-Based Nav Item Array

**What:** Derive a nav items array from `useRole()` flags, filter to accessible items only, and map to nav components.

**When to use:** For role-aware navigation where items are hidden (not grayed out) if user lacks access.

**Example:**
```typescript
// Inside AppLayout or Sidebar
import { useRole } from "@/context/AuthContext";
import { useLocation } from "react-router";
import { LayoutGrid, ClipboardList, CheckSquare } from "lucide-react";

const ALL_NAV_ITEMS = [
  { label: "Builder", path: "/admin/builder", icon: LayoutGrid,    requiresAdmin: true },
  { label: "Fill Forms", path: "/worker/forms", icon: ClipboardList, requiresWorker: true },
  { label: "Reviews", path: "/reviewer/dashboard", icon: CheckSquare, requiresReviewer: true },
];

function useNavItems() {
  const { isAdmin, isWorker, isReviewer } = useRole();
  return ALL_NAV_ITEMS.filter(item => {
    if (item.requiresAdmin && !isAdmin) return false;
    if (item.requiresWorker && !isWorker) return false;
    if (item.requiresReviewer && !isReviewer) return false;
    return true;
  });
}
```

Note: `useRole()` in `AuthContext.tsx` already maps admin → isWorker + isReviewer, so admin users will naturally see all three items via the filter above.

### Pattern 4: User Identity via Clerk `useUser()`

**What:** Clerk's `useUser()` hook returns `user.firstName`, `user.lastName`, `user.imageUrl`. Use `<img>` with `imageUrl` and fall back to initials if null.

**When to use:** Anywhere user identity is displayed. This hook is available directly from `@clerk/clerk-react` which is already installed.

**Example:**
```typescript
// Source: Context7 /clerk/clerk-docs — useUser() hook
import { useUser } from "@clerk/clerk-react";

function UserIdentity() {
  const { user, isLoaded } = useUser();
  if (!isLoaded || !user) return null;

  const initials = [user.firstName, user.lastName]
    .filter(Boolean)
    .map(n => n![0])
    .join("");

  return (
    <div className="flex items-center gap-2">
      {user.imageUrl ? (
        <img src={user.imageUrl} alt="Avatar" className="w-8 h-8 rounded-full object-cover" />
      ) : (
        <div className="w-8 h-8 rounded-full bg-muted flex items-center justify-center text-sm font-medium">
          {initials || "?"}
        </div>
      )}
      <span className="text-sm font-medium">
        {[user.firstName, user.lastName].filter(Boolean).join(" ")}
      </span>
    </div>
  );
}
```

**Dev mode caveat:** In dev mode without credentials, `MockAuthProvider` is used and `useUser()` will return a Clerk loading state without a real user. The mock must be handled — either provide a fallback or use `useAuth()` from `AuthContext` for role data and only call `useUser()` when `isDevModeWithoutCredentials` is false.

### Pattern 5: Active Item Highlight with `useLocation()`

**What:** Compare current `location.pathname` to nav item path to set active styles.

**Example:**
```typescript
import { useLocation, Link } from "react-router";
import { cn } from "@/lib/utils";

function NavItem({ path, label, icon: Icon }) {
  const { pathname } = useLocation();
  const isActive = pathname === path || pathname.startsWith(path + "/");

  return (
    <Link
      to={path}
      className={cn(
        "flex items-center gap-3 px-3 py-2 rounded-md text-sm font-medium transition-colors",
        isActive
          ? "bg-primary text-primary-foreground"
          : "text-muted-foreground hover:bg-accent hover:text-accent-foreground"
      )}
    >
      <Icon className="w-5 h-5 shrink-0" />
      <span>{label}</span>
    </Link>
  );
}
```

### Pattern 6: Mobile Bottom Tab Layout Spacing

**What:** Mobile bottom tab bar is `position: fixed` at the bottom. Page content needs `pb-16 md:pb-0` (or similar) to prevent content from being obscured by the tab bar.

**When to use:** Always when using a fixed bottom nav on mobile. This is a CSS-only concern, no library needed.

### Pattern 7: OfflineBanner Placement

**What:** `OfflineBanner` is currently `fixed top-0 left-0 right-0 z-50`. On desktop, it sits above the persistent header. On mobile with a top bar, it also sits above it (same z-50 behavior). This is acceptable without changes — `fixed` elements are viewport-relative and overlay everything.

**Recommendation:** Keep `OfflineBanner` as-is (fixed, z-50). It will work correctly in the new layout. The `SyncIndicator` should move from the current inline header to either: the desktop top header (keep current position) or the desktop sidebar footer near user identity. Since desktop top header is retained, keep `SyncIndicator` there. On mobile, the SyncIndicator can be a compact icon-only variant in the mobile top bar or omitted to keep the top bar slim.

### Anti-Patterns to Avoid

- **Duplicating layout in every page component:** Don't add sidebar/bottom-tabs inside `AdminBuilder`, `FormFillingPage`, etc. The layout shell renders once in `AppLayout`.
- **Putting nav in `ProtectedRoute`:** `ProtectedRoute` handles auth gating, not layout. Keep them separate.
- **Using `window.location.pathname` for active state:** Use `useLocation()` from react-router — it's reactive and handles SPA navigation correctly.
- **Role guard inside nav items:** Nav items are filtered out entirely (hidden) — don't render them disabled/grayed. The role guards (`AdminRoute`, `WorkerRoute`, `ReviewerRoute`) on the routes themselves remain for security.
- **Calling `useUser()` in dev mode mock path:** When `isDevModeWithoutCredentials` is true, Clerk providers are not mounted. Calling `useUser()` will throw. Guard with the flag or provide a mock identity component.

## Common Pitfalls

### Pitfall 1: Dev Mode `useUser()` Crash

**What goes wrong:** When `isDevModeWithoutCredentials` is true, the app uses `MockAuthProvider` instead of `ClerkProvider`. Calling `useUser()` from `@clerk/clerk-react` outside a `ClerkProvider` throws an error.

**Why it happens:** `useUser()` requires a Clerk context that only exists inside `<ClerkProvider>`. In dev mock mode, there is no `ClerkProvider`.

**How to avoid:** Check `isDevModeWithoutCredentials` before calling `useUser()`, or create a wrapper hook:
```typescript
function useUserIdentity() {
  if (isDevModeWithoutCredentials) {
    return { name: "Dev User", imageUrl: null };
  }
  const { user } = useUser();
  return {
    name: [user?.firstName, user?.lastName].filter(Boolean).join(" ") || "User",
    imageUrl: user?.imageUrl ?? null,
  };
}
```

**Warning signs:** `Error: useUser can only be used within the <ClerkProvider> component` in development console.

### Pitfall 2: Mobile Content Obscured by Fixed Bottom Tab Bar

**What goes wrong:** Page content at the bottom of the scroll is hidden behind the fixed bottom tab bar (typically 56-64px high).

**Why it happens:** `position: fixed` elements don't affect document flow. The `<main>` area has no bottom padding.

**How to avoid:** Add `pb-16 md:pb-0` (or `pb-14`) to the `<main>` element. `pb-16` = 64px covers a standard 56px tab bar with margin.

**Warning signs:** Last form fields in `FormFillingPage` are unreachable on mobile.

### Pitfall 3: Desktop Redirect Logic Still Points to Hard-Coded Default

**What goes wrong:** `ProtectedRoute` redirects `/` to `/admin/builder` unconditionally (line in `protected.tsx`). With the new nav, a worker navigating to `/` still gets redirected to `/admin/builder` and sees an access-denied state.

**Why it happens:** The redirect in `ProtectedRoute.useEffect` uses `isAdmin` first, but if `isAdmin` is false and `isWorker` is true, it goes to `/worker/forms`. The logic exists but the catch-all `<Route path="*">` in `AppRoutes` also redirects to `/admin/builder`. This will redirect workers who hit an unknown path to an admin page.

**How to avoid:** Update the catch-all `<Route path="*">` redirect to be role-aware, or redirect to `/` and let `ProtectedRoute`'s logic handle it. Simplest fix: change `<Navigate to="/admin/builder" replace />` in catch-all to `<Navigate to="/" replace />` so `ProtectedRoute` handles role-based dispatch.

**Warning signs:** Worker role users see an empty/error screen after typing an unknown URL.

### Pitfall 4: No-Role User Gets Redirect Loop

**What goes wrong:** A user with `role: null` hits `/`, `ProtectedRoute` tries to redirect but none of `isAdmin/isWorker/isReviewer` is true, so nothing happens. The user stays on `/` which renders a `<Navigate to="/admin/builder" />` → 403 loop.

**Why it happens:** Current logic only handles the three known roles.

**How to avoid:** Add a `no-role` route or page. When `role === null` and user is authenticated, show the "Contact your admin" message directly in the nav shell. A simple `/` route component that checks role and renders the appropriate message covers this.

**Warning signs:** Newly invited users (before role assignment) see a blank page or redirect loop.

### Pitfall 5: SyncIndicator Dropdown Clipped by Sidebar

**What goes wrong:** `SyncIndicator` renders an absolutely positioned dropdown (`absolute top-full right-0`). If placed in the sidebar footer, the dropdown may be clipped by `overflow: hidden` on the sidebar container.

**Why it happens:** Absolutely positioned elements are clipped by ancestors with `overflow: hidden`.

**How to avoid:** Don't set `overflow: hidden` on the sidebar wrapper. Or keep `SyncIndicator` in the desktop top header (its current position) where the header has no overflow clipping.
